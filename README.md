# Приложение для проверки доступности URL

## 🛠 Стек технологий
- **Язык программирования:** Go
- **База данных:** PostgreSQL
- **Брокер сообщений:** RabbitMQ

## 🌐 API Endpoints

### `/checks`
| Метод | Описание |
|-------|----------|
| `GET`  | Получение всех записей проверяемых адресов |
| `POST` | Добавление нового URL для мониторинга |

### `/sites`
- **Метод:** `GET`
- **Параметры:** `url="name_url"`
- **Описание:** Получение статистики по конкретному сайту.
- **Формат ответа:** `JSON`, содержащий:
  - количество положительных ответов,
  - количество ошибок,
  - среднее время обработки каждого запроса по указанному сайту.

## ⚙️ Архитектура и рабочий процесс

1. **Планировщик (Scheduler)**
   - Периодически извлекает записи из базы данных.
   - Публикует каждую запись в основную очередь `sites_queue`.

2. **Воркеры (Workers)**
   - Запущено **3 экземпляра** воркеров.
   - Подписываются на `sites_queue` и выполняют проверку доступности URL.
   - **Успех:** Если URL доступен, проверка фиксируется как успешная.
   - **Ошибка:** Если URL недоступен, сообщение отправляется в отдельную очередь для повторных попыток (Retry Queue).
   - **Превышение лимита попыток:** Сайт помечается как `заблокированный`, а сообщение перемещается в `failed_queue`.

3. **Схема очередей (RabbitMQ)**
   - `sites_queue` — основная очередь задач на проверку.
   - `retry_queue` — очередь повторных попыток.
   - `failed_queue` — очередь окончательно заблокированных ресурсов.

## 📈 Агрегация статистики
Данные по каждому сайлу накапливаются на основе результатов работы воркеров. Эндпоинт `/sites` возвращает актуальную сводку в формате JSON для указанного домена.

## 🔮 Планы на развитие

- Расширение метрик и настройка гибких правил повторных попыток.
